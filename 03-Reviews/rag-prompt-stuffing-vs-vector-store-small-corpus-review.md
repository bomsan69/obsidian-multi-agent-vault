Strict review of [[rag-prompt-stuffing-vs-vector-store-small-corpus]] (consensus_score 92). 두 리뷰어(Claude, Codex)가 모든 정량적·사실적 주장을 검증했습니다. 결론: **핵심 방향(소규모 코퍼스엔 prompt 삽입이 우세)은 여전히 타당하지만, 정량적 임계값의 정합성과 몇몇 과장된 표현이 실제보다 확신을 부풀리고 있습니다.**

## 🔵 Claude — Reviewer Notes

**핵심 결론은 여전히 타당합니다.** 다만 strict 기준에서 짚어야 할 지점들:

- **한국어 특수성 누락:** "A4 3페이지 ≈ 2,000~4,000 토큰"은 사실상 Codex가 계산한 **영어 기준**(1,500~2,500단어) 추정치를 그대로 가져온 것입니다. 한글은 BPE 계열 토크나이저에서 음절당 토큰 소비가 영어 단어보다 비효율적인 경우가 많아, 실제 한국어 3페이지 문서는 표기된 범위보다 토큰 수가 더 많을 수 있습니다. 주제 자체가 한국어 문서인데 토큰 계산은 영어 기준을 그대로 썼다는 점에서 `[VERIFY]`.
- **Synthesis의 "수치가 겹친다"는 주장이 과장됨:** Claude는 정성적으로만, Codex는 "문서 50~150개/10만~30만 토큰", Gemini는 "수백 페이지"(3페이지 환산 시 대략 수십~백여 개 문서)라고 했는데, 이를 "같은 구간을 가리킨다"고 정리한 건 실제보다 정합성을 부풀린 서술입니다.
- **"Prompt caching이 비용 문제를 해소한다"는 과장:** 실제로는 지연시간 최대 80%, 입력 비용 최대 90% **절감**이지 "제거"가 아닙니다. 캐시 쓰기/읽기에도 별도 비용이 있습니다. `[VERIFY]`
- **Freshness 논리가 다소 뒤바뀜:** prompt 삽입은 매번 최신 텍스트를 통째로 보내므로 신선도가 항상 보장되고, 오히려 RAG가 재임베딩 트리거 누락 시 stale해질 위험이 있다는 점이 빠졌습니다.
- **코드:** `client.responses.create`/`.output_text`, `faiss.IndexFlatIP`+`normalize_L2` 조합은 정확합니다. 다만 `split_docs`/`docs` 미정의, `top_k` 초과·검색 결과 없음(`ids == -1`) 처리 누락으로 그대로 실행 불가능한 의사코드입니다.
- **Lost in the middle 인용은 정확** (Liu et al., arXiv:2307.03172).

## 🔴 Codex — Reviewer Notes

- **토큰/페이지 환산은 대략적 추정일 뿐:** "A4 3페이지 ≈ 한글 2,000~3,000자 ≈ 2,000~4,000토큰"은 성긴 한국어 산문엔 그럴듯하지만 일반화하기 어렵습니다. 폰트·여백·줄간격·표 포함 여부·토크나이저에 따라 크게 달라집니다. Codex의 "1,500~2,500단어" 표현은 한국어엔 특히 부정확합니다.
- **컨텍스트 윈도우 주장은 방향은 맞지만 부정확/구식:** GPT-4.1은 문서상 1,047,576 토큰, Claude는 모델별로 1M/200K, Gemini도 모델별 long-context를 지원 — "128K~1M"은 일부 모델군을 과소평가하고 "주류"를 과일반화한 표현입니다.
- **[VERIFY] 전환점 수치("1~20개 문서/5만 토큰 미만", "50~150개/10만~30만 토큰")는 독립적으로 뒷받침되지 않는 경험칙**입니다. 가격 정책, 캐시 적중률, 지연시간 목표, 질의량, 태스크 종류, 답변 길이, 검색 품질에 따라 달라집니다.
- **내부 불일치:** Claude는 "1개~수십 개"까지 괜찮다고 했지만 Codex는 문서 30개면 이미 6만~12만 토큰으로 비용·지연시간 부담이 시작된다고 했고, Gemini는 "2만 토큰 미만"을 직접 삽입 기준으로, "수백 페이지"를 임베디드 스토어 전환 기준으로 제시 — Synthesis가 이 수치들이 "겹친다"고 한 건 과장입니다.
- **"Prompt caching이 반복 전송의 비용·지연시간 페널티를 사라지게 한다"는 표현은 틀렸거나 과함:** OpenAI는 캐싱이 지연시간을 최대 80%, 입력 비용을 최대 90% **줄인다**고 명시하지 "제거"한다고 하지 않습니다. Claude의 캐시 쓰기/읽기도 별도 비용이 있고, Gemini의 "sub-second" 지연시간 주장도 근거가 없습니다.
- **"문서가 자주 바뀌면 RAG가 유리하다"는 서술은 오해의 소지가 있습니다.** Prompt 삽입은 항상 최신 텍스트를 그대로 사용하고, RAG는 증분 재임베딩이 정확해야 하며 그렇지 않으면 오히려 stale해질 수 있습니다. RAG는 신선도 자체보다 재처리 대역폭을 줄여주는 것에 가깝습니다.
- **Lost in the middle 인용은 정확:** Liu et al., "Lost in the Middle: How Language Models Use Long Contexts," arXiv:2307.03172 / TACL 2023.
- **코드:** `client.responses.create(...)`은 실제 OpenAI SDK/API 패턴이고 `.output_text`도 Responses API의 실제 편의 필드입니다. `faiss.IndexFlatIP` + 양쪽 벡터에 `faiss.normalize_L2` 적용은 코사인 유사도 검색 방식으로 올바릅니다(FAISS 공식 문서에도 정규화된 내적 = 코사인 검색으로 문서화됨).
- **코드 버그/공백:** `docs`, `split_docs`가 미정의. `k > len(chunks)`나 `ids == -1`에 대한 방어 코드 없음. 대량 `chunks`에 대한 배치 처리·rate-limit 대응 없음. 출처(메타데이터) 정보 없음. "이 컨텍스트만 사용하라"는 지시가 근거 없는 답변을 강하게 막아주는 보장은 아님.

## Action Items

- **[FIX] 토큰 계산:** 한국어 토큰화 특성(음절당 토큰 비효율)을 반영해 "A4 3페이지 ≈ 2,000~4,000 토큰"을 한국어 기준으로 재추정하거나, "영어 기준 추정치이며 한국어는 더 높을 수 있음"이라 명시.
- **[FIX] Synthesis 과장:** "세 관점의 구체적 수치가 겹친다"는 표현을 "방향은 일치하나 구체적 임계값은 모델마다 다르며 근거도 각기 다른 가정에 기반한 경험칙"으로 완화.
- **[FIX] Prompt caching 서술:** "비용·지연시간 문제가 사라진다" → "지연시간 최대 80%, 입력 비용 최대 90% 절감(제거 아님)"으로 정정, 캐시 쓰기 비용 존재 명시.
- **[FIX] Freshness 논리:** prompt 삽입 쪽이 항상 최신 상태를 보장하고, RAG는 재임베딩 누락 시 오히려 stale해질 수 있다는 방향으로 재작성.
- **[FIX] 코드 완성도:** `split_docs`/`docs` 정의 또는 "의사코드(pseudo-code)임" 명시, `top_k`/검색결과 없음 방어 코드 언급.
- **[VERIFY] 전환점 수치:** "1~20개/5만 토큰", "50~150개/10만~30만 토큰" — 특정 벤치마크나 논문 근거 없는 경험칙. 실사용 시 자체 비용·지연시간 측정으로 검증 필요.
- **[VERIFY] 컨텍스트 윈도우 일반화:** "128K~1M"이 일부 모델(예: 특정 Claude/Gemini 티어)의 실제 한도를 과소평가할 수 있음 — 사용할 모델의 공식 문서로 재확인 필요.

## Summary

**review_score: 74/100.** 핵심 아키텍처 결론(소규모 코퍼스=prompt 삽입, 대규모=vector RAG)과 "lost in the middle" 인용, 코드의 핵심 API 사용법(Responses API, FAISS 코사인 유사도)은 정확합니다. 점수를 깎은 요인은: (1) 한국어라는 주제 특성을 반영하지 못한 영어 기준 토큰 추정, (2) Synthesis가 실제보다 수치 정합성을 부풀린 것, (3) prompt caching을 "문제 해소"로 과장, (4) freshness 논리의 방향 오류, (5) 코드가 의사코드 수준이라는 점을 명시하지 않은 것. 모두 짧은 수정으로 해결 가능하며, 결론 자체를 뒤집는 수준은 아닙니다.
