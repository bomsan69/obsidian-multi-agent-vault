**실전 케이스:** 식당 C/S(고객 응대) 챗봇. 제약 조건:
① 정보는 한글/영문 모두 가능(다국어) ② 문서는 A4 3페이지 이내 ③ 정보 변경은 월 1회 ④ **문서에서 찾을 수 없는 질문에는 절대 답변 금지** ⑤ 반복적인 질문이 매우 많음.

기존 프레임(거버넌스 게이트 → 질의량 손익분기)을 이 케이스에 대입해 적합한 아키텍처를 토론했습니다.

## Round 1 — 입장

### 🔵 Claude
**입장: Prompt 삽입(전체 문서 삽입) + prompt caching + 강한 거부(refusal) 지시가 정답. Vector-store RAG는 오히려 위험하다.**

거버넌스 게이트 미해당(단일 식당, 사용자별 권한 차이 없음), 질의량 손익분기도 코퍼스가 극소(A4 3페이지)라 우리 리서치가 확립한 전환점(문서 50~150개/10만~30만 토큰)에 한참 못 미침 → 질의량이 아무리 많아도 전환 사유 아님.

**결정적 이유:** 제약 ④가 오히려 prompt 삽입을 요구합니다. Vector RAG의 실패 모드는 검색 누락(retrieval miss)이고, 이는 "문서에 없다"를 오판하거나 엉뚱한 청크로 그럴듯하게 지어내는 위험을 만듭니다. 전체 문서를 통째로 넣으면 이 실패 모드 자체가 없습니다. 3페이지짜리 문서에 청킹을 도입하는 건 최우선 요구사항(hallucination 절대 금지)에 불필요한 위험을 추가하는 것입니다.

실무안: 시스템 프롬프트에 문서 전체(한/영 원문) + 강한 거부 지시, prompt caching(월 1회 갱신이라 캐시 적중률 사실상 100%), 질문 언어 감지 후 같은 언어로 응답.

### 🔴 Codex
**입장: Full prompt-stuffing + 결정적 캐시 + 문서기반 abstention 게이트.**

검색 누락이 ④에서 치명적임을 강조. 실무안: 시스템 프롬프트에 전체 KO/EN 문서 + "제공 문서에 명시된 내용만 답하라, 없으면 거절하라, 추정·일반상식·외부지식 금지"라는 강한 규칙. 메뉴명·시간·정책·알레르기 항목은 구조화 JSON으로도 병행해 자유서술 문서에만 의존하지 않게 함. ⑤(반복 질의)는 vector-store 도입 사유가 아니라 **캐시 도입 사유** — 문서 버전 해시 + 정규화된 질문 + 언어 코드로 exact cache. 월 1회 갱신 시 버전 해시 변경으로 캐시 일괄 무효화.

### 🟡 Gemini
**입장: Full Prompt-Stuffing.**

다국어 교차 검색(cross-lingual retrieval)은 언어 간 임베딩 공간 불일치로 품질이 떨어지는 반면, prompt 삽입은 대형 LLM의 다국어 in-context 추론을 그대로 활용 가능. Prompt caching으로 입력 비용 90%+ 절감, Redis/GPTCache 같은 인메모리 캐시로 완전 동일 질의는 LLM 호출 자체를 생략해 비용·지연시간을 거의 0으로.

**Round 1 결과:** 세 참가자 모두 독립적으로 동일한 아키텍처(full prompt-stuffing + caching + 강한 거부 지시)에 도달.

## Round 2 — 반박 (프로덕션 배포 관점에서 맹점 공격)

### 🔵 Claude
1. **지시 준수는 100% 보장 안 됨.** "가정해보자면" 같은 우회 질문에 모델이 지시를 어길 위험. → **2단계 검증** 필요: 답변 생성 후, 같은 문서를 근거로 "이 답변이 실제로 문서에 명시돼 있는가"를 별도 확인(또는 구조화 출력으로 근거 문장을 강제 동반, 근거 없으면 자동 거부 응답으로 대체).
2. **날짜/시간 의존 질문("오늘 영업하나요?")이 캐시 히트율 100% 주장을 깹니다.** 정적 문서와 동적 컨텍스트(날짜·실시간 상태)를 분리해, 동적 정보는 캐시 안 되는 부분에 넣어야 함.
3. **멀티 식당 SaaS 확장 시** 테넌트별 문서 격리가 필수가 되어 거버넌스 게이트가 재발동하지만, 여전히 벡터 스토어가 아니라 테넌트별 완전 분리(독립 프롬프트/캐시 키)로 충분.

### 🔴 Codex
1. 시스템 프롬프트 지시만으로는 부족 — **2단계 게이트** 필요: 1차(LLM이 `supported=true/false` + 근거 필드 ID 반환) → 2차(서버가 근거 ID가 실제 문서/구조화 데이터에 존재하는지 검증, 없으면 답변 폐기하고 고정 거절문 반환). "모델에게 거절 지시"가 아니라 **애플리케이션 레벨에서 근거 없는 답변 출고 금지**.
2. 날짜/시간 주입은 "거의 100% 캐시 히트" 주장에 caveat를 만듦 — 정적 문서 prefix를 먼저 고정하고 동적 블록은 뒤에 붙여야 함. 웨이팅 같은 실시간 데이터는 애초에 prompt-stuffing 문제가 아니라 POS/대기시스템 연동 문제.
3. 수백 개 식당 SaaS 확장 시 tenant isolation, 문서 버전 관리, 캐시 네임스페이스 분리, cross-tenant response cache leakage가 핵심 리스크 — vector RAG는 여전히 불필요하지만 `tenant_id + doc_version` 기반 격리와 배포 승인 workflow는 필수.
4. "절대 문서 밖 답변 금지"는 지시만으로 100% 달성 불가 — 현실적 목표는 "모델 단독 보장"이 아니라 **출력 검증/근거 검증/고정 fallback을 포함한 시스템 보장**.

### 🟡 Gemini
1. **캐싱과 동적 변수의 충돌:** 정적 문서를 캐시 세그먼트 상단 접두사로 고정하고, 동적 컨텍스트·유저 질문은 캐시 체크포인트 이후(suffix)에 배치해야 캐시 효율이 보존됨.
2. **멀티테넌트 확장 시 새로운 리스크(Claude/Codex와 다른 관점):** 수백 개 식당으로 확장되면 각 테넌트 문서들이 LLM 제공사의 **공유 캐시 메모리를 경쟁적으로 점유**해 빈번한 캐시 무효화(eviction)·스래싱(thrashing)이 발생할 수 있음 — "100% 캐시 적중" 주장이 무색해지고 오히려 고비용·지연이 생길 위험.
3. Prompt 지시 단독으로는 jailbreak/prompt injection을 100% 방어 못함.
4. **결정적 사후 검증 가드레일:** 모델이 답변 시 원문 내 출처 구절(quote)을 JSON 필드로 강제 출력 → 백엔드에서 해당 구절이 실제 문서에 존재하는지 문자열 일치(exact match) 검증, 또는 가벼운 NLI 모델로 entailment 검증하는 **이중 게이트(dual-gate)** 없이는 프로덕션 신뢰도 달성 불가.

## ✅ Consensus Synthesis

두 라운드 모두에서 세 모델이 독립적으로 강하게 수렴했습니다. 1라운드는 아키텍처 방향(prompt 삽입), 2라운드는 그 방향을 프로덕션 수준으로 구체화하는 3가지 보완점에 도달했습니다:

**최종 권장 아키텍처:**

1. **1차 기준 아키텍처:** Vector-store RAG가 아니라 **Full prompt-stuffing** — 문서 전체(한글+영문 원문)를 시스템 프롬프트에 포함. 이 규모(A4 3페이지)에서 청킹·검색은 이득 없이 검색 누락(retrieval miss) 리스크만 추가하므로, 제약 ④(hallucination 절대 금지)와 정면으로 충돌합니다.
2. **근거 검증 이중 게이트(세 모델 만장일치, 이번 토론의 핵심 산출물):** 시스템 프롬프트의 "거부 지시"만으로는 불충분합니다. 프로덕션 수준에서는 (a) 모델이 답변과 함께 **문서 내 근거 문장/필드 ID를 구조화 출력으로 강제 반환**하고, (b) 서버가 그 근거가 실제 문서에 존재하는지 **검증**(문자열 일치 또는 경량 NLI)한 뒤, 근거 없으면 **고정 거절 응답으로 강제 대체**해야 합니다. "모델이 알아서 거부하길 기대"하는 게 아니라 "애플리케이션이 근거 없는 답변의 출고 자체를 막는" 구조입니다.
3. **캐싱 설계의 정교화:** "월 1회 갱신 + 반복 질문 다수"는 prompt caching에 이상적인 조건이지만, **정적 문서(캐시 prefix)와 동적 정보(현재 날짜/시간, 실시간 상태 — suffix)를 분리**해야 합니다. 그렇지 않으면 "오늘 영업하나요?" 같은 흔한 질문이 캐시를 매번 깨뜨립니다. 실시간 웨이팅 같은 정보는애초에 문서 기반이 아니라 별도 시스템(POS/대기 연동) 문제입니다.
4. **다국어 처리:** 별도 임베딩 기반 교차언어 검색보다, 문서 자체를 한/영 병기로 컨텍스트에 넣고 질문 언어를 감지해 같은 언어로 답하게 하는 것이 이 규모에서 더 안전하고 간단합니다.
5. **향후 확장(멀티 식당 SaaS) 시 재검토 필요:** 규모가 커져도 문서당 3페이지가 유지된다면 vector RAG는 여전히 불필요하지만, 테넌트 격리(문서·캐시 네임스페이스 분리)가 새로운 요구사항으로 추가됩니다. Gemini는 여기에 더해 **공유 캐시 자원 경쟁으로 인한 스래싱** 위험도 지적했습니다 — Claude/Codex가 강조한 "격리 필요성"과는 다른 각도의 유효한 추가 리스크입니다.

**결론:** 이 식당 C/S 챗봇에는 **vector-store RAG가 필요 없습니다.** 다만 "prompt 삽입 + 시스템 프롬프트 지시"만으로는 제약 ④를 프로덕션 수준으로 보장할 수 없고, **근거 검증 이중 게이트**를 반드시 추가해야 합니다. 이게 이번 토론이 원본 리서치/디베이트에 추가하는 새로운 결론입니다.
