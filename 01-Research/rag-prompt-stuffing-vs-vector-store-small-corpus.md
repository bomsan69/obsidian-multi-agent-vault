## 🔵 Claude — Overview & Architectural Framing

"RAG를 만든다"는 질문 앞에 먼저 물어야 할 것은 **"검색(retrieval)이 정말 필요한가"**입니다. 코퍼스 규모가 이 질문의 답을 결정합니다.

**규모 감각 잡기.** A4 3페이지는 한글 기준 약 2,000~3,000자입니다. 아래 토큰 수치(2,000~4,000)는 **영어 단어 수 기준 추정치를 그대로 가져온 것**이며, 한글은 BPE 계열 토크나이저에서 음절당 토큰 소비가 영어 단어보다 비효율적인 경우가 많아 실제 한국어 문서는 이보다 더 많은 토큰(대략 4,000~8,000 수준까지도)을 소비할 수 있습니다. 그럼에도 현재 주류 모델(Claude, GPT, Gemini)의 컨텍스트 윈도우는 128K~1M 토큰이므로, **문서 하나, 심지어 수십 개를 합쳐도 프롬프트에 통째로 넣는 게 기술적으로 전혀 무리가 없습니다.** 즉 "3페이지짜리 문서 하나"를 위해 벡터 스토어를 만드는 건 대부분의 경우 **과잉 엔지니어링**입니다.

**두 접근의 아키텍처적 차이:**

- **Prompt 삽입(context stuffing):** 매 호출마다 문서 전체를 시스템/유저 메시지에 넣습니다. 인덱싱 파이프라인이 없고, 코드가 단순하며, "최신 내용"이 항상 정확히 반영됩니다(재임베딩 지연이 없음). 단점은 호출마다 같은 토큰을 반복 전송해 **비용이 선형으로 늘고**, 문서 수가 늘면 결국 컨텍스트 한계에 부딪힙니다.
- **Embedding + 벡터 스토어:** 문서를 청크로 쪼개 임베딩하고, 질의 시 관련 청크만 상위 k개를 골라 프롬프트에 넣습니다. 코퍼스가 커져도(수백~수만 문서) 확장되고, 호출당 비용이 낮게 유지됩니다. 단점은 **청킹 경계 오류, 임베딩 품질, top-k 튜닝** 같은 새로운 실패 지점이 생기고, 인덱싱 파이프라인(및 재인덱싱 트리거)이라는 운영 부담이 붙습니다.

**결정 기준 (아키텍처 관점):**

- **문서 1~수십 개, 각각 몇 페이지 수준(이번 케이스):** prompt 삽입이 정답에 가깝습니다. 복잡성 대비 이득이 없습니다. 특히 prompt caching(Claude/Gemini/OpenAI 모두 지원)을 쓰면 반복 전송 비용 문제도 크게 줄어듭니다.
- **문서가 수백~수만 개로 늘거나, 전체 코퍼스가 컨텍스트 윈도우를 넘어서는 시점:** 이때부터 벡터 스토어가 유일하게 확장 가능한 선택지가 됩니다.
- **중간 지대(수십~수백 문서, 총량이 컨텍스트에 들어가긴 하지만 크다):** "가벼운 검색"(예: BM25 키워드 검색이나 요약 인덱스)이 벡터 스토어보다 먼저 시도해볼 만한 대안입니다.

**경험칙:** "검색을 얼마나 정교하게 할까"보다 먼저 "지금 코퍼스 크기에서 검색이 필요하긴 한가"를 물으세요. 이번처럼 3페이지 수준의 소규모 문서라면, 벡터 스토어는 이르며, prompt 삽입으로 시작해 **코퍼스가 실제로 커질 때** 검색 계층을 추가하는 점진적 전략이 더 낫습니다.

## 🔴 Codex — Technical Depth

A4 3페이지 문서는 (영어 기준) 보통 단어 수 1,500~2,500개, 토큰으로는 약 2천~4천 토큰 수준입니다. **이 수치는 영어 기준이며, 한국어 문서는 토크나이저 특성상 더 많은 토큰을 소비할 수 있어 그대로 적용하기 어렵습니다.** 그럼에도 최신 long-context 모델(128K 토큰은 흔하고, OpenAI/Anthropic급 API엔 1M 토큰 윈도우도 존재 — 다만 이는 모델·티어에 따라 다르며 모든 모델이 1M을 지원하는 것은 아닙니다) 기준으로는 여전히 작은 양입니다 ([OpenAI GPT-4.1](https://openai.com/index/gpt-4-1/), [Claude 문서](https://platform.claude.com/docs/en/about-claude/models/overview)). 그래서 **문서 하나**만 다룬다면, 전체 텍스트를 그대로 넣는 게 청킹도, 임베딩 드리프트도, 검색 누락도 없어 보통 더 단순하고 정확도도 높습니다.

전환점은 코퍼스 규모입니다. 대략적인 계산:

- `문서 1개 ≈ 2천~4천 토큰`: 그대로 넣기 충분히 사소함.
- `문서 30개 ≈ 6만~12만 토큰`: 128K 윈도우에서 프롬프트/출력이 작을 때만 가능; 이 지점부터 지연시간·비용이 부담되기 시작.
- `문서 100개 ≈ 20만~40만 토큰`: 128K 모델 다수엔 너무 크고, 1M 모델에서도 비용 부담.
- `문서 250개 ≈ 50만~100만 토큰`: 1M 모델에서도 실질적 한계에 근접.
- `문서 1,000개 ≈ 200만~400만 토큰`: 통째로 넣기는 더 이상 실행 불가능.

비용·지연시간의 확장 방식도 다릅니다. Prompt 삽입은 **매 질의마다** 코퍼스 전체의 입력 비용을 지불합니다 — 매번 30만 토큰을 보낸다면 모든 질문이 30만 토큰짜리 프롬프트가 됩니다. Vector RAG는 임베딩/인덱싱 비용을 한 번만 지불하고, 이후 매 질의는 질문만 임베딩한 뒤 `top_k=5` 정도의 청크(각 500~1,000 토큰), 즉 3천~8천 토큰 정도만 가져옵니다. 반복 질의가 많은 상황에서는 이쪽이 대체로 훨씬 저렴하고 빠릅니다.

정확도 측면에는 상반된 실패 양상 두 가지가 있습니다. Long-context 삽입은 검색 누락은 없지만 **"lost in the middle"** 현상을 겪을 수 있습니다 — 관련 사실이 긴 입력 깊숙이 있으면 모델이 이를 덜 신뢰성 있게 활용할 수 있습니다 ([Liu et al.](https://arxiv.org/abs/2307.03172)). RAG는 주의(attention)를 좁혀주지만, 잘못된 청크 경계·약한 임베딩·부실한 메타데이터 필터·잘못된 `top_k` 설정으로 실패할 수 있습니다.

신선도(freshness) 측면은 오히려 prompt 삽입 쪽이 유리합니다 — 매 호출마다 최신 전체 텍스트를 그대로 보내므로 항상 최신 상태가 보장됩니다. RAG는 문서가 바뀔 때마다 해당 청크를 정확히 재임베딩해야 하며, 이 트리거가 누락되거나 지연되면 오히려 **stale한 인덱스로 답변하는 위험**이 생깁니다. RAG의 이점은 신선도 자체보다는 "변경분만 재처리하면 되는 낮은 재처리 대역폭"에 가깝습니다.

**경험칙:** 문서 1~20개 또는 코퍼스 총량 5만 토큰 미만이면 prompt 삽입을 쓰세요. 문서 50~150개, 코퍼스 10만~30만 토큰 이상, 반복 질의가 많거나, 갱신이 잦거나, 지연시간·비용이 중요해지는 시점부터 vector-store RAG를 우선 고려하기 시작하세요.

> ⚠️ 아래는 **의사코드(pseudo-code)**입니다. `docs`, `split_docs`는 정의돼 있지 않고, `top_k`가 청크 수를 초과하거나 검색 결과가 없는 경우(`ids == -1`)에 대한 방어 코드도 없습니다. 실제 사용 전 이 부분들을 채워 넣어야 합니다.

```python
from openai import OpenAI
import faiss, numpy as np

client = OpenAI()
chunks = split_docs(docs, tokens_per_chunk=700, overlap=100)  # split_docs는 직접 구현 필요

emb = client.embeddings.create(
    model="text-embedding-3-small",
    input=chunks,
).data
vecs = np.array([e.embedding for e in emb], dtype="float32")

index = faiss.IndexFlatIP(vecs.shape[1])
faiss.normalize_L2(vecs)
index.add(vecs)

def answer(question, k=5):
    q = client.embeddings.create(
        model="text-embedding-3-small",
        input=[question],
    ).data[0].embedding
    q = np.array([q], dtype="float32")
    faiss.normalize_L2(q)

    _, ids = index.search(q, k)
    context = "\n\n".join(chunks[i] for i in ids[0])

    return client.responses.create(
        model="gpt-4.1-mini",
        input=f"Use only this context:\n{context}\n\nQuestion: {question}",
    ).output_text
```

## 🟡 Gemini — Ecosystem, Variants & Serving

2024~2025년 LLM 생태계는 소규모 코퍼스 RAG의 트레이드오프를 근본적으로 바꿔 놓았습니다. 각각 A4 3페이지 정도인 소수의 짧은 문서(총 2만 토큰 미만)라면, 도구 선택은 하드웨어 한계가 아니라 **운영 복잡도와 API 성능**에 달려 있습니다.

**1. 벡터 DB 없이 (Prompt 삽입 + Caching)**
초소규모 코퍼스에서는 벡터 데이터베이스를 아예 건너뛰는 것이 생태계의 지배적 흐름입니다. 최신 모델(Gemini 2.x/3.x, Claude 3.5, GPT-4o)은 이 정도 분량을 넉넉한 컨텍스트 윈도우 안에서 무리 없이 처리합니다. 결정적으로, **prompt caching**(Anthropic·Google·OpenAI 모두 지원)이 보편화되면서 같은 정적 컨텍스트를 반복 전송하던 기존의 비용·지연시간 페널티가 크게 줄었습니다 — 예를 들어 OpenAI는 캐시 적중 시 지연시간 최대 80%, 입력 비용 최대 90% **절감**을 명시합니다(페널티가 "사라지는" 것은 아니며, 캐시 쓰기/읽기 자체에도 별도 비용이 있습니다). 원시 API 호출이나 최소한의 오케스트레이션(예: 표준 LangChain/LlamaIndex 문서 로더)만으로 가장 민첩하고 인프라가 필요 없는 경로가 됩니다.

**2. 경량 임베디드 벡터 스토어**
코퍼스가 동적이거나 다소 확장될 것으로 예상되면(예: 수십 개 문서까지), 실무자들은 가볍고 서버리스인 엔진 쪽으로 기웁니다. **sqlite-vec**, **LanceDB**, 인메모리 **Chroma**나 **FAISS** 같은 임베디드 옵션이 여기에 정확히 들어맞습니다. 프로세스 내부에서 실행되고, 외부 인프라가 전혀 필요 없으며, 애플리케이션과 함께 버전 관리가 가능하고, 네트워크 왕복 오버헤드도 없습니다.

**3. 매니지드 벡터 DB (Pinecone, Qdrant, Weaviate)**
짧은 문서 몇 개 수준에서 매니지드 벡터 데이터베이스를 도입하는 것은 명백한 과잉 엔지니어링 안티패턴입니다. 컬렉션 스키마 유지보수, 네트워크 지연, 매니지드 서비스 비용 같은 운영 부담이 이 규모에서는 아무런 이득도 주지 않습니다.

**생태계 결론:** 원시 텍스트를 네이티브 API의 prompt caching과 함께 프롬프트에 직접 로드하는 것부터 시작하세요. 이는 인프라 없이도 가장 높은 의미론적 정확도를 냅니다. 코퍼스가 수백 페이지를 넘어서기 시작할 때만 `sqlite-vec`나 `LanceDB` 같은 임베디드·제로컨피그 스토어로 전환하세요.

## ✅ Consensus Synthesis

세 모델의 결론이 방향 면에서는 강하게 수렴합니다. 다만 **구체적 수치까지 정확히 겹치는 것은 아니며**, 각 모델의 추정치는 서로 다른 가정에 기반한 경험칙이라는 점을 짚어야 합니다:

1. **규모 감각의 방향은 일치합니다.** A4 3페이지는 (영어 기준 추정으로) 2천~4천 토큰 수준이며, 한국어는 이보다 클 수 있습니다. 어느 쪽이든 현재 모델의 컨텍스트 윈도우(128K~1M, 모델별 상이)에 비해 작습니다. (Claude·Codex 합의, 단 정확한 수치는 언어·모델에 따라 달라짐)
2. **소규모(문서 1~수십 개)에서는 prompt 삽입이 우세합니다.** 청킹·임베딩 드리프트·검색 누락이 없고, prompt caching 덕분에 반복 전송 비용 부담도 상당히 줄어듭니다(단, 완전히 사라지지는 않음). (세 모델 모두 합의 — Gemini는 이를 "생태계의 지배적 흐름"이라 표현)
3. **전환점의 정확한 수치는 모델마다 다릅니다.** Codex는 "문서 50~150개 / 코퍼스 10만~30만 토큰"부터 vector RAG 우위 시작을 제시하고, Gemini는 "수백 페이지"(3페이지/문서 환산 시 대략 수십~백여 개 문서 — Codex 수치보다 다소 낮은 구간)를 임베디드 스토어 전환 시점으로, Claude는 정성적으로 "코퍼스가 실제로 커질 때"라고만 표현합니다. **세 수치 모두 벤치마크가 아닌 경험칙이며, 방향(작으면 prompt 삽입, 크면 RAG)은 일치하지만 정확한 임계값을 이 노트만으로 확정할 수는 없습니다** — 실사용 시 자체 비용·지연시간 측정으로 검증이 필요합니다.
4. **실패 양상은 서로 다른 종류입니다.** Prompt 삽입은 "lost in the middle"(긴 입력 속 정보 활용 저하) 위험이, RAG는 청킹 경계·임베딩 품질·top-k 튜닝 관련 위험이 있습니다. (Codex)
5. **생태계 단계론도 일치합니다.** 벡터 DB 없음 → 임베디드 경량 스토어(sqlite-vec, LanceDB, Chroma, FAISS) → 매니지드 벡터 DB(Pinecone 등) 순으로, 이번 케이스(문서 몇 개)는 명확히 1단계에 속합니다. 매니지드 벡터 DB는 이 규모에서 "안티패턴"이라는 표현까지 나왔습니다. (Gemini)

**결론:** 이번처럼 A4 3페이지 규모 문서 몇 개를 다루는 RAG라면, **벡터 스토어 없이 prompt에 문서를 직접 삽입하고 prompt caching을 활용하는 것**이 세 모델 모두가 도달한 답입니다. 벡터 스토어 도입은 코퍼스가 실제로 수십~수백 문서 이상으로 늘어나는 시점까지 미루는 것이 합리적입니다.

**케이스:** 한/영 혼용, A4 3페이지 이내 문서, 월 1회 갱신, **문서 밖 답변 절대 금지**, 반복 질문 다수.

**결론 (세 모델 만장일치, 2라운드 걸쳐 검증됨):**

1. **Vector-store RAG 불필요.** 이 규모(A4 3페이지)에서 청킹·검색은 이득 없이 검색 누락(retrieval miss) 리스크만 추가 — 이는 "문서 밖 답변 절대 금지"라는 최우선 요구사항과 정면 충돌합니다. **Full prompt-stuffing**(문서 전체를 시스템 프롬프트에 포함)이 정답입니다.
2. **단, "시스템 프롬프트 지시만으로는 부족"— 근거 검증 이중 게이트가 필수:** 모델이 답변과 함께 문서 내 근거 문장/필드 ID를 구조화 출력으로 강제 반환하고, 서버가 그 근거가 실제 문서에 존재하는지 검증한 뒤, 근거 없으면 고정 거절 응답으로 강제 대체해야 합니다. "모델이 알아서 거부하길 기대"가 아니라 "애플리케이션이 근거 없는 답변의 출고 자체를 막는" 구조가 필요합니다.
3. **캐싱은 정적 문서(prefix)와 동적 정보(오늘 날짜·실시간 상태, suffix)를 분리** 설계해야 "오늘 영업하나요?" 같은 흔한 질문이 캐시를 깨뜨리지 않습니다.
4. **다국어는 문서 자체를 한/영 병기로 컨텍스트에 넣는 것**이 별도 교차언어 임베딩 검색보다 이 규모에서 더 안전합니다.
5. 향후 멀티 식당 SaaS로 확장 시 테넌트별 문서/캐시 격리가 새로운 요구사항으로 추가되지만(문서당 3페이지가 유지된다면) vector RAG는 여전히 불필요합니다.
